## 1. C++ 기반 탐색 기능 모듈 개발

### 주요 담당 업무

- 다중 경로 탐색 및 경로 안내 시작 기능 개발
- 현재 경로 기반 재탐색 기능 개발
- 주기적 재탐색 및 사용자 요청 재탐색 기능 개발
- 목적지 및 경유지 편집 기능 개발
- 경로 옵션 관리 기능 개발
- 탐색 화면 및 정보 패널 상태 관리
- 지도 연동 데이터 처리
- UI 이벤트 처리 및 탐색 상태 관리

### 주요 기능

#### 경로 탐색

- 다중 경로 탐색 결과를 기반으로 경로 선택 화면 표출
- 사용자 선택에 따른 경로 안내 시작 처리
- 현재 경로 기준 재탐색 및 경로 업데이트 처리
- 재탐색 유형에 따라 화면 표출 여부와 경로 갱신 정책 분기 처리

#### 경로 편집

- 목적지 설정
- 경유지 추가, 삭제 및 순서 변경
- 경로 편집 결과에 따른 탐색 요청 처리
- 경로 편집 화면과 탐색 상태 연동

#### 경로 옵션 관리

- 경로 옵션을 역할별로 분리하여 관리
- 상위 옵션 관리 구조를 통해 하위 옵션들을 통합 관리
- 옵션 저장, 조회, 변경 기능 개발
- 옵션 변경에 따른 탐색 요청 및 상태 갱신 처리
- 지역, 플랫폼, 타입별 기능 분기 처리
- 화면 표시용 옵션과 실제 요청용 옵션을 분리하여 관리
- 신규 옵션 추가가 용이하도록 구조 개선

#### 탐색 화면 및 상태 관리

- 탐색 응답 데이터를 기반으로 UI 표현용 상태 객체 구성
- 탐색 상태 변경에 따라 UI가 자동 갱신되는 구조 구현
- 경로 선택 화면, 경로 편집 화면, 경로 옵션 화면, 정보 패널 상태 관리
- 탐색 상태를 기반으로 이벤트 처리 및 화면 상태 관리

#### 지도 연동

- 탐색 및 안내 시작 시점의 경로 표시 데이터 제어
- 목적지 및 경유지 표시 데이터 제어
- 안내 중 경로 변경에 따른 경로 표시 데이터 갱신 처리
- 지도 표시 계층과의 인터페이스를 통해 탐색 데이터 전달
- 지도 이동, 뷰 전환 등은 별도 모듈과 역할을 분리하고 탐색 데이터 제공에 집중

## 2. Kotlin 기반 탐색 기능 모듈 개발

### 개발 배경

기존 C++ 기반 모듈을 Kotlin 기반 모듈로 전환하는 구조를 검토하고 개발하였다.

기존 UI 구조는 유지하면서 Controller 계층을 Kotlin으로 대체하는 방향이었기 때문에, UI와 Controller 간의 책임 경계를 유지하면서 Kotlin의 비동기 처리와 상태 관리 방식을 적용하는 것이 중요했다.

### 주요 담당 업무

- Kotlin 기반 탐색 기능 모듈 초기 구조 설계 및 기능 구현
- 동기 API를 Coroutine 기반 비동기 호출 구조로 래핑
- API 응답 데이터를 내부 표현 모델로 변환
- 탐색 응답 데이터와 Controller 데이터를 분리한 계층 구조 설계
- StateFlow 기반 상태 관리 적용
- Polyline, 거리, 지도 표시 영역 계산 등 데이터 변환 기능 개발

### 구조 설계 및 구현 내용

- 초기 C++ 스타일 구조를 Kotlin 특성에 맞게 책임 분리 구조로 개선
- UI, API 요청, 응답 데이터 변환, 상태 전달 책임을 분리하여 구조 설계
- 탐색 응답 데이터와 Controller 데이터를 분리하여 확장 가능한 계층 구조 설계
- API 응답 데이터를 UI 표현에 적합한 내부 모델로 변환
- StateFlow 기반으로 상태 변경을 전달하는 구조 구현
- 플랫폼 데이터 변환 기능 개발
  - Polyline 데이터 변환
  - 거리 계산
  - 지도 표시 영역 계산

※ Kotlin 기반 탐색 기능 모듈은 초기 구조 및 주요 기능 개발까지 진행

## 문제 해결 경험

### 1. 책임 경계와 상태 관리 구조 개선

탐색 기능 개발 과정에서 가장 어려웠던 부분은 기능 구현 자체보다 모듈 간 책임 경계를 정의하는 것이었다.

탐색 응답 데이터, Controller, UI 계층이 동일한 데이터를 직접 공유할 경우 데이터 소유권이 불명확해지고, 상태 불일치 및 유지보수 비용 증가 문제가 발생할 수 있다고 판단하였다.

이를 해결하기 위해 탐색 응답 데이터를 Single Source of Truth로 유지하고, UI 표현을 위한 Controller 상태를 별도로 구성하였다.

이를 통해 탐색 응답 데이터와 UI 표현 데이터를 분리하고, 상태 변경에 따라 UI가 자동 갱신되는 구조를 구현하였다.

### 2. 경로 옵션 관리 구조 개선

기존 경로 옵션은 지역, 플랫폼, 타입에 따라 관리 위치와 방식이 분산되어 있어 신규 옵션 추가 및 유지보수에 어려움이 있었다.

이를 개선하기 위해 경로 옵션을 역할별로 분리하고, 상위 옵션 관리 구조에서 하위 옵션들을 통합 관리하도록 구조를 정리하였다.

또한 화면 표시용 옵션과 실제 요청용 옵션을 분리하여 관리함으로써 UI 표현과 요청 데이터의 책임을 명확히 구분하였다.

이를 통해 지역별, 플랫폼별, 타입별 옵션 추가 및 변경에 대한 유지보수성을 개선하였다.

### 3. 화면 전환 시 UI 깜빡임 문제 개선

화면 갱신 주기와 상태 반영 타이밍 불일치로 인해 화면 전환 시 UI 패널 깜빡임 문제가 발생하였다.

프로젝트 구조상 화면 갱신은 모듈 데이터 조회, 이벤트 처리, UI 데이터 전달 순서로 진행되었고, 이로 인해 이벤트 처리 이전 상태가 현재 화면에 반영되는 문제가 발생할 수 있었다.

문제 분석 후 화면 전환 전에 필요한 상태를 선반영하도록 처리 흐름을 개선하였다.

이를 통해 화면 전환 시 발생하던 UI 깜빡임 문제를 제거하고 화면 안정성을 향상시켰다.
